# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Context
Production-grade marketing + private-console website for MLAI Corporation. **Next.js 15 (App Router) + React 19, run on Bun** — one process serves pages AND the `/api/*` route handlers (WorkOS AuthKit, protected LLM/billing APIs, inquiry storage, telemetry). The previous Vite SPA, Bun/Hono server, and Rust/Axum migration plan are abandoned. Do not extend or restore those stacks; active work belongs in the TypeScript/TSX Next.js surface.

## Essential Commands
- `bun install` — Install dependencies (Bun 1.4+; `bun.lock` text lockfile)
- `bun run dev` — Next.js dev server, port 3000 (pages + `/api/*` in one process; `./start_dev.sh` is now just this)
- `bun run build` — **Runs `bun run sitemap` first** (so detail-page URLs never drift), then `next build`.
- `bun run start` — Production server (`next start`, port 3000); needs a build first.
- `bun run lint` — Type-check (`tsc --noEmit`); the primary gate.
- `bun run test` — Vitest smoke suite ([src/__tests__/content.test.ts](src/__tests__/content.test.ts)): validates the content against the Zod schema and asserts every blog/research/team slug is unique + URL-safe with a non-empty body.
- `bun run sitemap` — Regenerate `public/sitemap.xml` from the content layer ([scripts/generate-sitemap.ts](scripts/generate-sitemap.ts)).
- `bun run crawl` — Real-browser link-integrity check ([scripts/link-crawl.py](scripts/link-crawl.py) via the playwright-cli wrapper): BFS-crawls every route on :3000, click-tests each unique internal link, HEAD-checks asset links, fails on 404s/console errors. Needs `bun run dev` running (or `BASE_URL=` a deployed origin).

## Architecture

### Single-process model (the key thing to understand)
Next.js serves pages and `/api/*` route handlers from one process in both dev and prod — no proxy, no port split. Route handlers live under [app/api/](app/api/) and share server logic from `src/lib/server/` (session, db, workos, rate-limit, llm).

### Frontend (`app/` + `src/`)
- **App Router**: route files live in `app/` (one `page.tsx` per route, dynamic `[slug]` dirs for blog/research/team/products). Each route's `page.tsx` is a thin server component that exports `metadata` (from [src/lib/route-meta.ts](src/lib/route-meta.ts)) and renders the real view from [src/views/](src/views/) through a **per-route `client.tsx`** (e.g. `app/links/client.tsx` is just `"use client"; export { Links } from "@/views/Links"`) — that file is the `"use client"` boundary. Follow this exact pattern when adding a route: view in `src/views/`, `app/<route>/client.tsx` re-export, `app/<route>/page.tsx`, a `routeMetadata` entry, and a sitemap static entry. Cinematic surfaces + the TF demo cross via `next/dynamic` `ssr:false` — they touch browser APIs.
- **`src/views/` was `src/pages/`** — renamed because Next treats `src/pages/` as the Pages Router. Don't recreate `src/pages/`.
- **react-router compat shim**: the view components still import from `"react-router-dom"`, which is aliased (tsconfig `paths` + `next.config.ts` webpack/turbopack alias) to [src/lib/router-compat.tsx](src/lib/router-compat.tsx) — Link/NavLink/useLocation/useNavigate/useParams/useSearchParams over next/navigation. The npm `react-router-dom` package is removed; don't re-add it. The shim's `useLocation` deliberately avoids `useSearchParams()` (Suspense/prerender constraint) — pages that need live search params (Login) are wrapped in `<Suspense>` in their route file.
- Provider stack + page chrome (Navbar/Footer/skip-link/decorations) live in [app/providers.tsx](app/providers.tsx) (was main.tsx + Layout.tsx): `QueryClientProvider` → `AuthProvider` → `UIProvider`.
- Per-route SEO is data-driven from [src/lib/route-meta.ts](src/lib/route-meta.ts) (static map + blog/research/team/product slug derivers) via each route file's `metadata`/`generateMetadata`.
- API calls go through typed wrappers in [src/lib/api.ts](src/lib/api.ts), not ad-hoc `fetch` (except auth redirects).
- Path alias `@/*` → `src/*` (tsconfig paths; Next resolves it natively).

### Backend (`app/api/` + `src/lib/server/`)
- Every endpoint is a Next route handler under `app/api/.../route.ts`, sharing logic from `src/lib/server/`: `session.ts` (iron-session sealed cookie `mlai_session`, 7-day TTL, IP+UA-bound), `db.ts` (SQLite via `node:sqlite` `DatabaseSync`), `workos.ts` (AuthKit client, admin allowlist, MFA factor checks, `ADMIN_REQUIRE_MFA` fail-closed gate), `rate-limit.ts` (in-memory fixed windows), `llm.ts` (Gemini or safe scaffold).
- Surface: `/api/auth/login|signup|callback|me|logout|features|verify-user|mfa-status`, `PATCH /api/profile`, `/api/llm/status|chat`, `/api/billing/plans|checkout`, `/api/inquiries` (public rate-limited POST, admin-gated GET), `/api/telemetry` (allowlisted events, DNT/Sec-GPC honored, no identifiers stored) + `/api/telemetry/summary`.
- Auth degrades gracefully when `WORKOS_API_KEY`/`WORKOS_CLIENT_ID`/`SESSION_SECRET` are unset (redirect with `?error=auth_not_configured`). Login/signup redirects are 307 under Next (were 302) — same behavior for GET redirects.

### Retired stacks
Historical Vite, Hono, and Rust/Axum work is not part of the active product direction. Do not wire retired entrypoints, servers, or migration artifacts back into the app. If old code is needed for reference, use git history or `.retired/`; implement current behavior in Next route handlers and TypeScript/TSX modules.

### Telemetry, MFA, and WDBX V2 assets
- **Telemetry (privacy-respecting):** `POST /api/telemetry` accepts allowlisted events only (`inquiry_open/submit/success/close`), honors DNT/Sec-GPC with a 204 no-store, and persists ONLY event + pathname + timestamp to the `telemetry_events` table in `inquiries.db` — no IP/UA/user id (the client IP touches only the in-memory rate limiter). Client side: `src/lib/telemetry.ts` (`track()`, sendBeacon + keepalive fallback, checks doNotTrack/globalPrivacyControl before sending); hooks live in `ui-context.tsx` (open/close) and `InquiryForm.tsx` (submit/success). `GET /api/telemetry/summary` (admin allowlist + optional MFA gate) returns per-event counts and the open→success conversion rate. Keep new events allowlisted on both sides; never add identifying fields.
- **MFA:** policy is configured in the WorkOS Dashboard ([docs/mfa-workos-runbook.md](docs/mfa-workos-runbook.md) has the steps + lockout-safe rollout order). Code side: `GET /api/auth/mfa-status` lists enrolled factors; `ADMIN_EMAILS` allowlists admin readers; `ADMIN_REQUIRE_MFA=true` makes the admin reads (`GET /api/inquiries`, `GET /api/telemetry/summary`) fail closed (403) without an enrolled factor. The SDK's `listUserAuthFactors` exists at runtime but not in the bundled types — `src/lib/server/workos.ts` uses a narrow structural cast (`FactorListing`); don't "fix" it back to a direct call until the SDK types catch up.
- **WDBX V2 docs/PDFs:** the real Markdown documentation set from the sibling `wdbx` repo is mirrored at `public/docs/wdbx/*.md` (9 files incl. `limitations.md`) and surfaced in the Docs page `#wdbx-v2` section; research-paper PDFs (browser print-to-PDF of the live pages) live at `public/research/*.pdf`. The release note is the `wdbx-v2-release` blog entry. Re-mirror from `~/wdbx/docs/` when V2 docs change.
- **HeroScene micro-interactions:** `HeroScene.webgpu.tsx` (the active hero) has pointer parallax, hover brightening, and a click-triggered "backtrace pulse" (sparks travel inward along the spokes; core flashes on arrival) — all gated off under `prefers-reduced-motion`. Pulse flight starts OUTSIDE the core disc on purpose (additive sparks have no contrast against the bright core).

### Content data layer
Site copy is a single source of truth at [src/data/index.ts](src/data/index.ts), which aggregates per-domain modules in `src/data/categories/*` (about, platform, industries, services, research, blog, team, stats, faq, products, changelog) into a `content` object, with Zod schemas in [src/data/schemas.ts](src/data/schemas.ts). `changelog` drives the `/changelog` release-history page ([src/views/Changelog.tsx](src/views/Changelog.tsx)) — entries are milestone markers aligned to documented project history, and the page says so; `/links` ([src/views/Links.tsx](src/views/Links.tsx)) is the navigation hub of verified internal routes + external repos/docs. `products` drives the `/products/:slug` deep-dive pages ([src/views/Product.tsx](src/views/Product.tsx) — ABI Framework + Abbey, with KaTeX equations and the interactive demos in `src/components/demos/`); product routes are emitted into the sitemap and resolved in route metadata like the other dynamic slugs. Edit content there, not inline in components. (Note: older docs referenced a flat `src/data/content.ts` — that file no longer exists.)

- **Long-form content (blog + research) is data, not JSX.** Both `blog` and `research.publications` items carry a `slug` and a structured `body: BlogSection[]` (shared `BlogSectionSchema` of `{ heading?, paragraphs[], list?, math?, code? }`). `math` is an array of LaTeX strings rendered as KaTeX block equations (via [src/components/Math.tsx](src/components/Math.tsx); KaTeX CSS imported in `app/layout.tsx`); `code` is an array of `{ lang?, file?, code }` blocks rendered monospace by `ArticleSections`. To add an article/paper, append a fully-bodied entry to the data module; do not hardcode prose in a component. List cards (`Blog.tsx`, `Research.tsx`, `Home.tsx`) link by `slug`.
- **The detail pages share one shell.** [BlogPost.tsx](src/views/BlogPost.tsx) (`/blog/:slug`), [ResearchPaper.tsx](src/views/ResearchPaper.tsx) (`/research/:slug`), and the founder profile all render through [src/components/article.tsx](src/components/article.tsx): `ArticleLayout` (full reading shell), `ArticleSections` (the heading/paragraph/list/math body renderer), and `ArticleNotFound` (in-page 404). Research/blog tag→class colors live in [src/lib/tag-colors.ts](src/lib/tag-colors.ts), not inline.
- **Per-route SEO is data-derived.** [src/lib/route-meta.ts](src/lib/route-meta.ts) maps static paths to `<title>`/description and derives metadata for `/blog/:slug`, `/research/:slug`, `/team/:slug`, and `/products/:slug` from the matching `content` entry. Unknown paths and invalid slugs resolve to a `noindex` 404 meta (not the default org meta). New indexable routes need an entry here.
- **Content claims are constrained.** Per the sibling `abi` repo's `docs/contracts/external-claims-audit.md`, external collateral (this site included) must NOT cite specific QPS/latency/accuracy/energy/model-benchmark numbers, distributed sharding, AES/RBAC, regulatory certifications, or non-existent stacks unless a repo artifact proves them. Frame metrics as **targets/goals** (as `src/data/categories/stats.ts` already does) and ground new copy in verifiable architecture (the real `abi` CLI, MCP tools, WDBX stores, Abbey/Aviva/Abi persona routing).

### Styling
TailwindCSS v4 via PostCSS (no `tailwind.config.js`; tokens/utilities live in [src/index.css](src/index.css)). Custom design system: typography scale `--text-h0`→`--text-h5` (Major-Third), and utility classes `.container-custom` (max-w-7xl page width), `.container-prose` (~68ch reading measure), `.section-y` (responsive vertical rhythm `py-20 md:py-24 lg:py-28`), `.section-title`, `.glass-card`, `.label-chip`. Dark theme is set via `:root` tokens; the `.dark` class only sets `color-scheme: dark` so it won't override the MLAI palette. Animation via Framer Motion; icons from Lucide React; primitives from `@base-ui/react` + shadcn-style wrappers in `src/components/ui/`.

- **Brand = "Signal" / Indigo.** **Indigo primary** + **sky/ice secondary** on a slate near-black canvas; **fuchsia** is a rare "trace" accent. (Palette history: blue/cyan → emerald/teal → indigo/sky — do **not** reintroduce emerald/teal/blue/cyan for brand color.) Color is driven by OKLCH semantic tokens in `:root` (`--primary: oklch(0.58 0.18 270)`, `--accent`, `--ring`, `--background`, …) re-exposed to Tailwind via `@theme inline` (shadcn Tailwind-v4 structure). There is **also** a legacy `@theme` hex block (`--color-primary: #6366f1`, `--color-bg: #0a0b12`, …) driving `bg-bg`/`text-text-dim`/`bg-primary`; **update both on a re-skin.** **Re-value tokens to re-skin; don't restructure the plumbing.** `--primary-foreground` is **light** (the indigo primary is mid-dark, L≈0.58) to keep ≥4.5:1 WCAG AA.
- **Components use Tailwind's `indigo`/`sky`/`fuchsia` ramps.** Migrate a re-skin with `sed 's/<old>-/<new>-/g'` over `src/` (BSD sed has no `\b` — use plain substrings). Hardcoded brand hex/rgb lives in the 3D hero glow (`HeroScene.tsx`), WDBX charts (`Dashboard.tsx`/`BarChart.tsx`: `WDBX_COLOR`), the persona accent dots in `Docs.tsx`, the SVG assets (`public/*.svg`), `index.html` (theme-color/mask-icon), and `manifest.webmanifest` — the utility-class sed misses these, so update them by hand.
- **Glass depth** is unified: both `.glass-card` (utility) and the shadcn `Card variant="glass"` ([src/components/ui/card.tsx](src/components/ui/card.tsx)) use a 1px inset top-edge highlight + indigo hover ring + lift. Keep them in sync if you tune one.
- **Logo/wordmark is a reusable component** ([src/components/Logo.tsx](src/components/Logo.tsx)) — node-graph mark on an indigo→sky gradient, used by Navbar + Footer + Login. Don't re-inline logo markup.
- **Fonts:** body/sans is self-hosted **Geist** (`@fontsource-variable/geist`); display **Outfit** + mono **JetBrains Mono** load async (non-render-blocking print→all swap) from `index.html`. Don't re-add a render-blocking Google Fonts `@import` in CSS.

## Deployment
Multi-stage [Dockerfile](Dockerfile) (`oven/bun` base) targeting Google Cloud Run; the runner executes `next start` on the injected `PORT`. Required secrets: `WORKOS_API_KEY`, `WORKOS_CLIENT_ID`, `SESSION_SECRET` (generate with `openssl rand -base64 32`); optionally `GEMINI_API_KEY`, `STRIPE_PAYMENT_LINK`, `APP_URL`/`FRONTEND_URL`, `ADMIN_REQUIRE_MFA`. Copy `.env.example` → `.env` for local setup. Keep production LLM keys server-side only.

These govern the *content* of the benchmark visualizations, which mirror sibling-project releases:

## Repository hygiene
- The working tree periodically accumulates rotating scratch/experiment files at the repo root (`*.json`/`*.txt`/`debug_*.ts` — e.g. past ones like `accent_usage.txt`, `metadata.json`, `massive_content.json` that have since been removed). None are imported by `src/`, `app/`, or the build — verify with a quick `grep` before trusting any root-level scratch file; treat them as throwaway, not source of truth, and don't wire them into the app.
- `src/views/TFPoseDemo.tsx` (route `/tf-pose-demo`) is the only consumer of the heavy `@tensorflow/tfjs` + `@tensorflow-models/posenet` dependencies. It's a standalone demo, not part of the marketing/console product — keep it isolated so it doesn't pull TF into the main bundle path.
